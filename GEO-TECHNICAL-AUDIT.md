## Critical Issues — fix this week

### 🔴 1. Sitemap missing the two newest articles

`/sitemap.xml` lists 12 URLs and the `lastmod` on every URL is stuck at `2026-03-16`. Two recently-shipped articles are missing:

- `/research/rise-of-agentic-economy` (March 27, 2026 — by Carson)
- `/research/state-of-compliance-2026` (April 26, 2026 — by Sarah, just shipped)

Both articles return 200 and are fully prerendered, but **AI crawlers and search engines that discover via sitemap will not find them**. This directly costs citations on the article that the team just spent a session distributing.

The sitemap is a static file at `public/sitemap.xml`. It should be auto-generated by the build pipeline so new articles + freshly-edited routes update `<lastmod>` automatically.

**Fix (sketch):**
```ts
// scripts/generate-sitemap.ts — run from prerender.mjs after ROUTES is loaded
import { ARTICLES } from '../src/data/articles';
const urls = [
  { loc: '/', priority: 1.0, lastmod: today },
  { loc: '/research', priority: 0.8, lastmod: latestArticleDate },
  ...ARTICLES.map(a => ({
    loc: `/research/${a.slug}`,
    priority: 0.7,
    lastmod: a.date,
  })),
  { loc: '/whitepaper', priority: 0.7, lastmod: '2026-03-15' },
  { loc: '/roadmap', priority: 0.6, lastmod: today },
  // ... contact / privacy / terms
  { loc: '/docs', priority: 0.5, lastmod: today },
  { loc: '/docs/quickstart', priority: 0.5, lastmod: today },
  { loc: '/docs/api', priority: 0.5, lastmod: today },
  // ... rest of /docs/* + /wiki
];
// emit XML to dist/sitemap.xml during build
```

This also adds the 12 docs/wiki URLs that are currently missing entirely from the sitemap. Net coverage goes from 12 URLs → ~24 URLs.

### 🔴 2. Static assets have no long-term cache headers

Every static asset returns:
```
cache-control: public, max-age=0, must-revalidate
```

Including content-hashed bundles like `/assets/index-CeLfL5gV.js` whose filename guarantees they never change. This forces every browser revalidation round-trip. Vercel CDN cache (`x-vercel-cache: HIT`) covers the first hop, but every repeat visitor's browser still talks back to the CDN for every file.

**Fix in `vercel.json`:**
```json
{
  "headers": [
    {
      "source": "/assets/(.*)",
      "headers": [
        { "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }
      ]
    },
    {
      "source": "/(.*)\\.(woff2|webp|png|svg|ico|jpg|jpeg)",
      "headers": [
        { "key": "Cache-Control", "value": "public, max-age=2592000" }
      ]
    }
  ]
}
```

For repeat visitors, this is a meaningful LCP/INP improvement on every page after the first.

## Warnings — fix this month

### 🟡 3. IndexNow not implemented

IndexNow is supported by **Bing, Yandex, Seznam, Naver**, and notably **ChatGPT uses Bing's index**. Without IndexNow, new articles take days to appear in Bing → ChatGPT.

`/.well-known/indexnow-key.txt` returns 404. No CMS-side IndexNow integration detected.

**Fix:** generate a UUID, write it to `public/<key>.txt`, and POST a JSON payload to `https://api.indexnow.org/indexnow` from the build pipeline whenever an article is added. ~30 minutes of work.

### 🟡 4. JS bundle is single-chunk (547 KB uncompressed, 163 KB Brotli)

Vite's build output already warns about this:
> Some chunks are larger than 500 kB after minification. Consider using dynamic import() to code-split.

The wire weight (Brotli) is fine. The parsing/execution cost in the browser scales with uncompressed size. Code-splitting by route would let the homepage skip downloading + parsing the article-page React tree, the docs tree, the whitepaper tree, etc.

**Fix:** convert each top-level page in `src/App.tsx` to `React.lazy(() => import('./pages/...'))` with a `<Suspense fallback={...}>` boundary. Vite picks up the dynamic imports automatically and emits per-route chunks.

### 🟡 5. Mobile tap targets under 44×44 px

47 of 58 tappable elements (81%) are smaller than the WCAG 2.1 AA minimum of 44 × 44 CSS pixels. Most are footer links and the inline-link Wikipedia chips. This is a real mobile UX issue — fat fingers misclick.

**Fix:** in `src/components/Layout.tsx`, increase footer link `py-` padding so each link reaches 44 px tall. For inline chips/links inside body copy, the rule isn't strictly enforced (inline text), but isolated small icons/buttons (Linkedin / Twitter / copy share) need `min-h-[44px] min-w-[44px]`.

### 🟡 6. LCP not measured on prerendered pages

Browser perf API reports `largest-contentful-paint = 0` on the homepage. This isn't an actual LCP failure — it just means the LCP candidate (likely the headline text) rendered so fast that the API didn't fire. Real-user data via CrUX would clarify.

**Fix:** install Vercel Speed Insights or Cloudflare Web Analytics RUM (not Google Analytics — both privacy-respect and CrUX-compatible). Real LCP/INP/CLS measurements at the 75th percentile would replace the estimation here.
